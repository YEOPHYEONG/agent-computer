# V0.1.1 Improvement Plan

This document collects post-preview feedback and splits it into separate improvement tracks.

The goal is to discuss and improve one track at a time instead of mixing unrelated changes into one large update.

## Summary

Feedback points:

1. Agent routing should work naturally on every user request, not only at the beginning of a session.
2. Agent Computer needs an HTML/webpage builder agent for requests such as "turn this into a webpage" or "make an HTML landing page."
3. Research quality should be higher, especially for deep research.
4. Agents should ask better questions, clarify intent, and use human-in-the-loop checkpoints for large or ambiguous work.
5. Deep research should use the benchmark report as a design reference.
6. Agent Builder should optionally create Codex subagent playbooks when a new agent would benefit from parallel work.

## Improvement Tracks

| Track | Name | Core Question | Priority |
|---|---|---|---|
| A | Always-On Agent Routing | How does every meaningful request get routed to the right agent or chain, even mid-conversation? | High |
| B | Human-In-The-Loop Clarification | When should agents ask questions, and how do they avoid asking too many? | High |
| C | Deep-Dive Researcher Upgrade | What research architecture makes deep research substantially better? | High |
| D | HTML / Web Page Builder Agent | How should Agent Computer produce polished webpages, not only reports and PPT decks? | High |
| E | Agent Builder + Subagent Playbooks | How should new agent apps include Codex-specific parallel subagent instructions when useful? | Medium |
| F | Router / Chain Checkpoints | How should multi-agent chains coordinate questions and approvals without annoying the user? | Medium |

## Track A: Always-On Agent Routing

### Problem

Agent Computer can feel like it routes well at session start, but less reliably after a conversation has already begun.

Users may chat casually, then suddenly ask:

```text
Research this and make a PPT.
Turn this report into a webpage.
Organize this folder.
Draft an email to Alex.
```

Those requests should naturally trigger the correct agent or agent chain.

### Desired Behavior

For every user message, the runtime instructions should silently classify:

```text
Is this only a conversational answer?
Is this an agent task?
Does it need a project folder?
Does it need multiple agents?
Does it require clarification?
Does it require approval?
```

### Candidate Changes

- Update `AGENTS.md` and `CLAUDE.md` with an "always-on routing" rule.
- Strengthen `computer/agents/system/workspace-router/`.
- Add examples for mid-session routing.
- Add route smoke checks for natural mid-conversation requests.

### Files Likely Affected

- `AGENTS.md`
- `CLAUDE.md`
- `computer/agents/system/workspace-router/agent.md`
- `computer/agents/system/workspace-router/workflow.md`
- `computer/agents/system/workspace-router/output-template.md`
- `computer/tools/lib/registry.mjs`
- `computer/tools/lib/router.mjs` if present
- route smoke tests / release builder smoke prompts

### Open Questions

- Should every agent task explicitly mention the selected agent in the final answer?
- Should routing be visible by default, or only when the user asks?
- How aggressive should routing be for small tasks?

### Done When

- Mid-session work requests route correctly.
- "How do I use this?" still answers as Agent Computer usage.
- Simple conversational questions are not over-routed.
- External actions still require explicit approval.

## Track B: Human-In-The-Loop Clarification

### Problem

Agents sometimes proceed without fully understanding the user's intent, audience, constraints, or preferred output.

For high-stakes or high-effort artifacts, the user wants agents to ask useful questions and then move forward.

### Desired Behavior

Agents should use tiered clarification:

| Task Type | Behavior |
|---|---|
| Simple, reversible task | Make reasonable assumptions and proceed. |
| Ambiguous but low-risk task | Ask at most 1-2 clarifying questions or state assumptions. |
| Large artifact task | Clarify purpose, audience, format, depth, and constraints before major work. |
| Multi-agent chain | Ask consolidated checkpoint questions instead of each agent interrupting separately. |
| External action | Require explicit user approval. |

### Candidate Changes

- Add a shared `human-in-the-loop.md` policy.
- Update major agent workflows with an intake step.
- Add checkpoint language to `workspace-router`.
- Add templates for "assumptions if not answered."

### Files Likely Affected

- `computer/docs/human-in-the-loop.md`
- `AGENTS.md`
- `CLAUDE.md`
- `computer/agents/system/workspace-router/workflow.md`
- `computer/agents/work/deep-dive-researcher/workflow.md`
- `computer/agents/work/report-writer/workflow.md`
- `computer/agents/work/ppt-builder/workflow.md`
- `computer/agents/work/email-operator/workflow.md`
- future `web-page-builder`

### Open Questions

- What is the maximum number of questions per checkpoint?
- Should the default be "ask first" for research and presentation work?
- How should agents continue if the user does not answer?

### Done When

- Large tasks start with a concise intent checkpoint.
- Agents do not ask repetitive or low-value questions.
- Multi-agent chains consolidate questions.
- The final artifact records key assumptions.

## Track C: Deep-Dive Researcher Upgrade

### Problem

Research should feel meaningfully better than a quick web answer. It should ask better questions, go deeper, track evidence, verify citations, and adapt as findings emerge.

### Benchmark Reference

Use the internal/local benchmark report as a design reference:

```text
researcher-agent-deep-research-benchmark / best-researcher-agent-report-ko.md
```

This benchmark is a local design reference and is not included in the public release copy.

Key patterns to absorb:

- intake / research contract
- source policy
- Deep / Wide / Hybrid mode router
- N-pass question loop
- evidence store
- source ledger
- citation verifier
- claim verifier / QA
- dedicated worker notes for wide research
- synthesis writer separated from verification

### Desired Behavior

The researcher should decide among:

| Mode | Use When |
|---|---|
| Deep | One complex question needs causal, strategic, or interpretive depth. |
| Wide | Many independent items need the same rubric. |
| Hybrid | Broad coverage is needed first, followed by deep synthesis. |

### Candidate Outputs

```text
workspace/projects/<project-slug>/research/
  research-brief.md
  question-ledger.md
  source-ledger.md
  evidence-store.md
  worker-notes/
  synthesis.md
  citation-check.md
  research-qa.md
```

### Candidate Changes

- Upgrade `deep-dive-researcher` docs and workflow.
- Add Deep/Wide/Hybrid mode rules.
- Add source-ledger and question-ledger templates.
- Add citation and claim verification steps.
- Add route handoff expectations for report/PPT/web outputs.

### Files Likely Affected

- `computer/agents/work/deep-dive-researcher/README.md`
- `computer/agents/work/deep-dive-researcher/agent.md`
- `computer/agents/work/deep-dive-researcher/workflow.md`
- `computer/agents/work/deep-dive-researcher/output-template.md`
- `computer/agents/work/deep-dive-researcher/templates/`
- `computer/tools/lib/research.mjs`
- `computer/agents/system/qa-verifier/`

### Open Questions

- Should Wide mode be document-only at V0.1.1, or should it include executable helper scripts?
- Should citation verification be a separate `qa-verifier` mode?
- How much web/source count budget should default deep research use?

### Done When

- Deep research creates a research brief before final synthesis.
- Important claims have sources or uncertainty labels.
- Follow-up questions are captured in a question ledger.
- The researcher can explain why it chose Deep, Wide, or Hybrid.

## Track D: HTML / Web Page Builder Agent

### Problem

Users may ask for a webpage instead of a PPT:

```text
Turn this into an HTML page.
Make a webpage that explains this project.
Create a landing page from this report.
```

Agent Computer should handle that as naturally as PPT creation.

### Desired Behavior

Create a new work agent:

```text
computer/agents/work/web-page-builder/
```

Responsibilities:

- convert research/report/source material into webpage content structure
- preserve meaning and important detail
- create content spec and design spec
- build editable HTML/CSS/JS files
- preview locally when possible
- run screenshot/layout QA with Playwright or available browser tooling
- save outputs under:

```text
workspace/projects/<project-slug>/web/
workspace/projects/<project-slug>/assets/
workspace/projects/<project-slug>/qa/
```

### Candidate Changes

- Add `web-page-builder` agent app.
- Add web route detection to `workspace-router`.
- Add starter templates for landing page, report page, and microsite page.
- Add QA criteria for responsive layout, text overflow, broken links, and asset paths.

### Files Likely Affected

- `computer/agents/work/web-page-builder/`
- `computer/agents/README.md`
- `computer/system/agent-registry.md`
- `computer/system/tool-registry.md`
- `computer/agents/system/workspace-router/`
- `computer/tools/lib/`
- examples / route smoke checks

### Open Questions

- Should the first version generate static HTML only?
- Should CSS be inline, single `styles.css`, or framework-free?
- Should web outputs live under `web/` or `site/`?

### Done When

- A request for "make this an HTML webpage" routes to `web-page-builder`.
- The agent produces a project-local web folder.
- The output can be opened locally.
- Basic responsive/layout QA is documented or executed.

## Track E: Agent Builder + Subagent Playbooks

### Problem

Some agents benefit from parallel work, especially wide research, code review, benchmark comparison, and multi-item analysis.

Codex supports subagents, but official guidance says Codex should not spawn subagents automatically unless the user explicitly asks for subagents or parallel agent work.

### Source

OpenAI Codex docs:

```text
https://developers.openai.com/codex/concepts/subagents#triggering-subagent-workflows
```

Key guidance:

- Codex does not spawn subagents automatically.
- Use subagents only when explicitly requested.
- Subagent prompts need a bounded job, clear division of work, wait behavior, and return format.
- Subagents consume more tokens.

### Desired Behavior

`agent-builder` should optionally create Codex-specific playbooks:

```text
codex-subagents.md
parallel-worker-prompts.md
subagent-routing-notes.md
```

These files are execution recipes, not portable requirements.

### Candidate Changes

- Update `agent-builder` to ask whether the new agent needs parallel/subagent workflows.
- Add templates for subagent playbooks.
- Add warning that subagent use is Codex-specific and explicit.
- Add fallback instructions for non-subagent runtimes.

### Files Likely Affected

- `computer/agents/system/agent-builder/`
- `computer/docs/runtimes/codex.md`
- `computer/docs/runtimes/claude-code.md`
- `computer/templates/agent-app/`
- `computer/system/tool-registry.md`

### Open Questions

- Should subagent playbooks be generated for all agents or only selected agents?
- Should the user opt in during agent creation?
- How should Claude Code equivalents be documented, if any?

### Done When

- New agent apps can include optional Codex subagent instructions.
- The docs clearly say this is explicit, not automatic.
- Non-Codex runtimes have a sequential fallback path.

## Track F: Router / Chain Checkpoints

### Problem

In chained tasks, every agent asking its own questions can annoy users.

Example:

```text
Research this and make a PPT.
```

This may involve:

```text
workspace-router -> deep-dive-researcher -> report-writer -> ppt-builder -> qa-verifier
```

If each agent interrupts separately, the workflow feels fragmented.

### Desired Behavior

The router should consolidate questions into checkpoints:

1. Initial intent checkpoint
2. Research plan checkpoint if scope is broad
3. Artifact direction checkpoint before PPT/web/report production
4. Approval checkpoint before external actions or destructive file operations

### Candidate Changes

- Add chain-level checkpoint rules to `workspace-router`.
- Update agent workflows to defer non-urgent questions to the chain checkpoint.
- Add "assumptions if unanswered" pattern.

### Files Likely Affected

- `computer/agents/system/workspace-router/`
- `computer/docs/human-in-the-loop.md`
- `deep-dive-researcher`
- `report-writer`
- `ppt-builder`
- future `web-page-builder`

### Open Questions

- Should checkpoint questions be written to `workspace/projects/<slug>/tasks/`?
- Should the router ask questions before creating a project folder?
- How many unanswered assumptions are acceptable before proceeding?

### Done When

- Multi-agent tasks ask fewer, better questions.
- Checkpoints are visible in task artifacts when useful.
- The user can answer once and have the full chain use that context.

## Suggested Implementation Order

1. Track B: Human-in-the-loop policy
2. Track A: Always-on routing
3. Track F: Chain checkpoints
4. Track C: Deep-dive researcher upgrade
5. Track D: Web page builder
6. Track E: Agent Builder subagent playbooks

Reason:

- Clarification and routing rules affect every other improvement.
- Chain checkpoints prevent deep research, PPT, and web builder from asking redundant questions.
- Research quality upgrade is large and should use the benchmark report after the common policy is stable.
- Web page builder is a new feature and should inherit the same clarification/checkpoint rules.
- Subagent playbooks are powerful but runtime-specific, so they should come after the portable agent app model remains clear.
