# Image Deck Maker QA Checklist

## Source Boundary

- [ ] The deck uses approved source material only.
- [ ] Research, strategy, and factual synthesis were handled upstream when needed.
- [ ] Unsupported claims were not added for visual drama.

## Text Lock

- [ ] Every string intended for the generated images appears in `text-lock.md`.
- [ ] Korean text, names, numbers, URLs, dates, and claims were flagged.
- [ ] Required user approval was obtained before image generation.
- [ ] Default mode is `pure-imagegen`: visible slide text was rendered by `$imagegen`.
- [ ] If local text overlay was used, explicit `hybrid-overlay` approval is recorded.
- [ ] Text inside final images matches the approved text closely enough for use.
- [ ] Standard slides have enough visible text to communicate the main point without presenter narration.
- [ ] Standard slides are not merely title/subtitle mood images.
- [ ] Each slide has a content-fit structure chosen for its actual message.
- [ ] Text coverage targets are recorded and appropriate to the slide role.
- [ ] The deck does not force the same title/thesis/block structure onto every slide.

## Visual Quality

- [ ] Final slide visuals were generated with `$imagegen` / built-in `image_gen`.
- [ ] HTML/CSS/SVG/canvas/Sharp/browser rendering was not used as the primary creative generation engine.
- [ ] Any support renderer use was limited to approved hybrid overlays, contact sheets, package assembly, dimension checks, or QA.
- [ ] If `imagegen` was unavailable, the agent stopped instead of substituting a local renderer as the final generator.
- [ ] Slide count matches the contract.
- [ ] Aspect ratio matches the contract.
- [ ] Resolution is suitable for presentation use.
- [ ] Slides share one coherent design concept.
- [ ] No slide contains incoherent text, broken words, or unreadable labels.
- [ ] Important visual elements are not cropped or obscured.

## Package Quality

- [ ] PNG files are named in slide order.
- [ ] PPTX/PDF package exists when requested and opens.
- [ ] The final handoff states that the deck is image-based and mostly non-editable.
- [ ] Regeneration needs and known issues are documented.

## Hard Fail Conditions

- [ ] FAIL if final slide images were primarily generated by HTML/CSS/SVG/canvas/Sharp/browser rendering.
- [ ] FAIL if `$imagegen` / built-in `image_gen` is not recorded in the generation audit for a final image deck.
- [ ] FAIL if a project-local one-off renderer becomes the main creative engine instead of a support tool.
- [ ] FAIL if final slide text was added by local overlay without explicit `hybrid-overlay` approval.
- [ ] FAIL if the agent claims `pure-imagegen` while using renderer-added text.
- [ ] FAIL if most standard content slides are title/subtitle-only and do not contain meaningful supporting content blocks.
- [ ] FAIL if a rigid repeated slide structure is used where the content clearly calls for different visual/text structures.
- [ ] FAIL if text coverage is far too sparse for the approved source depth and slide purpose.
