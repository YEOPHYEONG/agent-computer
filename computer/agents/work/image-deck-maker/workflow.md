# Workflow

1. Confirm the requested artifact is an image-based deck.
   - If the user needs editable slides, route to `ppt-builder`.
   - If the user needs research first, route to `deep-dive-researcher` and `report-writer` before this agent.
2. Read the approved source package.
   - user brief
   - approved report
   - research handoff
   - brand guide
   - reference images
   - required text
3. Define the deck contract:
   - purpose
   - audience
   - use context
   - language
   - aspect ratio
   - slide count
   - output format
   - editability warning
   - source boundaries
   - approval requirements
   - production mode: `pure-imagegen` by default; `hybrid-overlay` only with explicit user approval
4. Write `deck-contract.md`.
5. Write `content-outline.md`.
   - slide number
   - slide role
   - message
   - exact approved text rich enough for the slide role
   - content-fit structure, chosen per slide
   - text coverage target, as an approximate percent range
   - visible text groups appropriate to the chosen structure
   - supporting source
   - must-not-say items
6. Write `design-outline.md`.
   - start with `Deck Design Concept`
   - then define each slide's visual composition
   - keep a consistent visual system across the deck
7. Write `text-lock.md`.
   - list every text string intended to appear in generated images
   - flag risky text: Korean, numbers, URLs, legal/financial wording, brand names
   - mark whether text will be rendered by `imagegen` or by approved `hybrid-overlay`
   - state the user approval gate
8. If text lock contains outcome-sensitive exact text, stop and wait for approval.
9. Write `image-prompts.md`.
   - one prompt per slide
   - aspect ratio
   - visual concept
   - exact text instruction for `$imagegen` to render inside the image
   - layout instructions matching the slide's content-fit structure
   - text coverage target and text hierarchy
   - negative prompt
   - filename
   - variants requested
10. Generate final slide visuals with `$imagegen` / built-in `image_gen`.
   - This is mandatory for final visual generation.
   - If `imagegen` is unavailable, stop and report the blocker. Do not silently substitute HTML/SVG/CSS/canvas rendering as the main generator.
   - In default `pure-imagegen` mode, visible slide text must be generated by `imagegen`.
   - If exact text is too risky for image generation, ask whether the user wants explicit `hybrid-overlay` mode. Do not switch silently.
11. Save images under:

```text
workspace/projects/<project-slug>/presentations/image-deck/generated/
```

12. Use support rendering only when needed:
   - locked text overlay only in approved `hybrid-overlay` mode
   - contact sheet
   - dimension/safe-area validation
   - package assembly
   - visual QA
13. Package generated slide images when possible:
   - PNG sequence
   - full-slide image PPTX
   - PDF export when supported
   - contact sheet
14. Run image-deck QA:
   - `imagegen` use confirmed
   - no HTML/SVG/CSS/canvas renderer used as the main creative engine
   - visible slide text generated by `imagegen` unless `hybrid-overlay` was explicitly approved
   - standard content slides are not title/subtitle-only
   - each slide's structure fits the content
   - text coverage target is suitable for the slide role and source depth
   - slide count
   - aspect ratio
   - resolution
   - exact text match
   - text legibility
   - missing/corrupt text
   - unsupported claims
   - consistency with design concept
   - final package opens
15. Save QA under:

```text
workspace/projects/<project-slug>/qa/<topic>_image-deck-qa.md
```

16. Final handoff must state:
   - `$imagegen` was used for final visual generation
   - whether production mode was `pure-imagegen` or approved `hybrid-overlay`
   - this is an image-based deck
   - most slide contents are not editable
   - which slides may need regeneration
   - where source outlines and prompts live
